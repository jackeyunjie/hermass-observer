## 2. 核心设计决策

### 决策 1：State 表达映射放在哪一层？

**方案 A：数据库视图层（推荐）**
- 创建 `v_perspective_state_human` 视图，在 SQL 中完成 Hex → 6命名/6节奏/共振度 的映射
- 优点：前端/AI/导出全部复用同一套逻辑；易审计
- 缺点：SQL 中写映射表有点冗长

**方案 B：后端 API 层**
- Python 字典映射，API 返回时转换
- 优点：灵活，可随时热更新
- 缺点：多处调用可能不一致

**方案 C：前端展示层**
- 前端 JS 字典映射
- 缺点：导出/AI 无法复用

**结论：方案 A + B 混合。**  
- 数据库视图给分析型查询用；
- 后端 API 给前端用（减少 SQL 复杂度）；
- 映射表抽离为独立 JSON 配置文件，前后端共同读取。

### 决策 2：AI Agent 集群用什么框架？

**方案 A：agently（潘杰已提到）**
- 已有探索，有监测数据
- 需要学习成本

**方案 B：Kimi Code CLI Agent 集群（本项目 Kimi 环境）**
- 用现有 Kimi agent 做提示词分工
- 适合快速验证

**方案 C：自研轻量路由**
- 一个 router 函数 + 多个 prompt 模板
- 最快实现，适合 P0

**结论：P0 用方案 C 快速上线，P1 迁移至 agently 或保留方案 C。**  
理由：一周时间不够学习新框架，先把提示词和路由逻辑跑通。

### 决策 3：缺失数据隐藏的实现方式

**前端方案（推荐）**
- 模板层判断 `{% if data %}...{% endif %}`
- 零后端改动，当天可上线

## 3. 数据层改造（零 Schema 变更）

### 3.1 新增视图：v_perspective_state_human

```sql
CREATE OR REPLACE VIEW v_perspective_state_human AS
SELECT
    state_date,
    symbol,
    name,
    
    -- 原始 Hex（保留）
    mn1_state_hex,
    w1_state_hex,
    d1_state_hex,
    ef_count,
    
    -- 6 命名映射（单个周期）
    CASE 
        WHEN CAST(mn1_state_hex AS INTEGER) IN (14,15) THEN '天时'
        WHEN CAST(mn1_state_hex AS INTEGER) IN (12,13) THEN '地利'
        WHEN CAST(mn1_state_hex AS INTEGER) IN (8,9,10,11) THEN '人和'
        WHEN CAST(mn1_state_hex AS INTEGER) IN (4,5,6,7) THEN '蓄力'
        WHEN CAST(mn1_state_hex AS INTEGER) IN (0,1,2,3) THEN '冬眠'
        WHEN CAST(mn1_state_hex AS INTEGER) < 0 THEN '逆位'
        ELSE '未知'
    END as mn1_state_name,
    -- 同理 w1_state_name, d1_state_name...
    
    -- 共振度（基于 ef_count + 方向）
    CASE
        WHEN ef_count = 3 THEN '🔥天时共振'
        WHEN ef_count = 2 THEN '☀️地利共振'
        WHEN ef_count = 1 THEN '🌤单一周期'
        WHEN ef_count = 0 AND (mn1_state_hex<0 OR w1_state_hex<0 OR d1_state_hex<0) THEN '⚡逆位共振'
        WHEN ef_count = 0 THEN '❄️无共振'
        ELSE '未知'
    END as resonance_level,
    
    -- 节奏（基于 D1 为主）
    CASE
        WHEN ef_count >= 2 AND CAST(d1_state_hex AS INTEGER) IN (14,15) THEN '生长季'
        WHEN ef_count >= 2 AND CAST(d1_state_hex AS INTEGER) IN (12,13) AND d1_days_extended > 10 THEN '秋收期'
        WHEN ef_count = 1 AND boundary_signal = '上突' AND boundary_distance <= 1.0 THEN '破土期'
        WHEN ef_count = 1 AND CAST(d1_state_hex AS INTEGER) IN (4,5,6,7) THEN '萌芽期'
        WHEN ef_count = 0 AND CAST(d1_state_hex AS INTEGER) < 0 THEN '逆风期'
        WHEN ef_count = 0 AND CAST(d1_state_hex AS INTEGER) IN (0,1,2,3) THEN '冬藏期'
        ELSE '过渡期'
    END as rhythm_phase,
    
    -- 建议动作（基于节奏）
    CASE
        WHEN rhythm_phase = '生长季' THEN '标准跟踪'
        WHEN rhythm_phase = '秋收期' THEN '防守/减仓'
        WHEN rhythm_phase = '破土期' THEN '跟踪确认'
        WHEN rhythm_phase = '萌芽期' THEN '观察'
        WHEN rhythm_phase = '逆风期' THEN '暂不推进'
        WHEN rhythm_phase = '冬藏期' THEN '等待'
        ELSE '观察'
    END as suggested_action
    
FROM d1_perspective_state
LEFT JOIN LATERAL (
    -- 需要确认 d1_days_extended 和 boundary_signal 是否存在
    SELECT 0 as d1_days_extended, '' as boundary_signal, 0.0 as boundary_distance
) AS extra ON true;
```

> **注意**：`d1_days_extended`、`boundary_signal`、`boundary_distance` 字段需要确认实际表结构是否存在。若不存在，节奏映射先简化为基于 `ef_count + d1_state_hex` 的版本。

### 6.2 判官 Agent 审核规则（代码实现）

```python
FORBIDDEN_WORDS = set(json.load(open("state_mapping.json"))["forbidden_words"])

def judge_review(text: str) -> dict:
    """返回审核结果，若发现禁用词则标记并建议替换"""
    found = [w for w in FORBIDDEN_WORDS if w in text]
    has_disclaimer = "不构成投资建议" in text
    return {
        "pass": len(found) == 0 and has_disclaimer,
        "forbidden_words": found,
        "missing_disclaimer": not has_disclaimer
    }
```

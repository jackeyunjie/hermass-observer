日期：2026-07-02
作者：KIMI

## 1. 产品结论（一句话）

**首页不做加法，只做减法：把 8 个平级模块收敛为 4 层递进阅读结构，让用户 5 秒扫完画面、15 秒找到关注点。**

## 2. 当前 8 个模块分析

### 2.1 逐一评估

| # | 模块 | 当前问题 | 判 |
|---|------|----------|----|
| 0 | **观象指令栏** | 占据首屏最上方，功能 OK | ✅ 保留 |
| 1 | **状态脉冲** | 数据摘要卡，提供全局锚点，有用 | ✅ 保留（轻量化） |
| 2 | **我的标的转折雷达** | 核心价值表，自选+候选+入口 | ✅ 保留（吸收时间窗矩阵） |
| 3 | **3D/3W/3M/6M 时间窗矩阵** | 只展示"选中标的"，没选中时空白，孤立感强 | ⚠️ 合并入雷达 |
| 4 | **经典策略信号灯** | 只占一行标签，权重刚好 | ✅ 保留（不变） |
| 4b | **全市场转折 Top** | 独立两列但实物感弱，缺概率数据支撑 | ⚠️ 合并入转折概率区 |
| 5 | **系统健康** | 放最底部是终端页模式，但当前展开太长 | ⚠️ 折叠到底部脚注 |

### 2.2 核心问题

当前 8 个模块是**平铺式排列**，用户从上滚到下看到 8 个独立卡片，缺乏递进阅读节奏：

```text
观象栏 → 脉冲 → 雷达 → 矩阵(空) → 信号 + Top → 健康
```

用户问题：「我的标的今天有什么变化？」需要滚动到第 3 个模块才能看到。

### 2.3 收敛方案

```text
Phase 2 首页四层递进：

┌─────────────────────────────────────────────────┐
│ L0: 观象指令栏（不变）                             │
├─────────────────────────────────────────────────┤
│ L1: 今日画面（状态脉冲 + 转折概览）                  │
│     一行摘要 + 三列小卡                            │
├─────────────────────────────────────────────────┤
│ L2: 我的标的（转折雷达 ← 合并时间窗矩阵）             │
│     表格 + 行内可展开时间窗                         │
├─────────────────────────────────────────────────┤
│ L3: 市场参考（全市场转折 Top ← 合并转折概率信号）      │
│     简洁列表 + 结构方向标签                         │
├─────────────────────────────────────────────────┤
│ 经典策略信号灯（保留，一行标签）                      │
├─────────────────────────────────────────────────┤
│ 系统健康（折叠到 footer-like 区域）                  │
└─────────────────────────────────────────────────┘
```

**变化总结**：

| 动作 | 模块 |
|------|------|
| ✅ 保留不变 | 观象指令栏、经典策略信号灯 |
| ✅ 保留轻量 | 状态脉冲 |
| ⚠️ 合并 | 时间窗矩阵 → 合并入我的标的雷达（行内可展开/折叠） |
| ⚠️ 吸收 | 全市场转折 Top → 吸收转折概率数据，变为「市场转折信号」（无裸数字） |
| 🔽 折叠 | 系统健康 → 默认折叠，点击展开 |
| 🆕 新增 | 「今日转折概览」迷你面板（3D/3W/3M/6M 统计摘要） |

## 3. 经典策略信号灯在首页的权重

**判断：当前权重已正确，不变。**

当前表现：
- 一行标签（`sentinel-row`），每个策略一个小卡片
- 无信号时显示「当前无经典策略规则信号」
- 有信号时：`VCP 规则信号 · 38` / `2560 风险信号 · 40` 等
- 点击跳转 `/mystrategies`（策略工坊页面的对应区）

为什么不变：
1. 大小刚好：一行标签不占空间，不抢注意力。
2. 隔离好：有独立免责说明「不参与 State 主系统判断」。
3. 用户心智对：看到信号→可点击→进入详情，不需要在首页展开。

**唯一建议**：信号标签目标链接从 `/mystrategies` 改为 `/sentinel`（哨兵总览页），因为 `/sentinel` 是专门为此设计的入口。

## 4. 转折概率进入首页的设计

### 4.1 核心风险

`turning_point_probability_latest.json` 中每个标的包含：

```json
{
  "prob_turn_up": 0.24,
  "prob_turn_down": 0.21,
  "prob_continue": 0.55,
  "turning_type": "uncertain",
  "confidence": 0.28,
  "evidence_items": ["D1 强势结构", "W1 方向偏空", ...],
  "risk_flags": ["低置信"]
}
```

**如果首页展示任何裸概率数字（如 24%），用户会自然读作交易信号。这是红线。**

### 4.2 安全展示规则

| 可以展示 | 不可展示 |
|----------|----------|
| 结构转强（高置信） | 上涨概率 62% |
| 结构转弱（低置信） | 下跌概率 38% |
| 持续结构（证据不足） | 综合评分 7.2 |
| 证据：D1 强势 + ADX 扩张 | 推荐关注 |
| 风险：W1 逆风 / 低置信 | 适合配置 |

### 4.3 展示位置：L3「市场转折信号」

替换当前「全市场转折 Top」，用转折概率数据增强：

```text
┌─────────────────────────────────────────────────┐
│ 市场转折信号                   仅作研究观察        │
├─────────────────────────────────────────────────┤
│ 标的         方向          证据             风险     │
│ 301163.SZ    结构转强      3 项证据       低置信    │
│              · D1 强势结构                        │
│              · ADX 扩张                           │
│              · BB 带宽释放                        │
│ 300553.SZ    持续结构      2 项证据       无        │
│ 002132.SZ    结构转弱      1 项证据       W1逆风    │
│ 002025.SZ    结构转强      4 项证据       无        │
└─────────────────────────────────────────────────┘
```

**映射规则**：

```python
# turning_type → 前端展示标签
LABEL_MAP = {
    "turn_up":    ("结构转强", "strong"),
    "turn_down":  ("结构转弱", "risk"),
    "continue":   ("持续结构", "neutral"),
    "uncertain":  ("证据不足", "muted"),
}
```

**置信度处理**：
- confidence >= 0.60 → 正常展示，不加标签
- 0.40 <= confidence < 0.60 → 右下角标注「中置信」
- confidence < 0.40 → 右下角标注「低置信」

**展示条数**：Top 5 per window（已有 `top_by_window`），优先展示 3W/3M（中期结构变化更有观察价值）。

### 4.4 为什么这样安全

1. **不展示任何百分比** → 用户无法把「结构转强」读成「62% 概率涨」。
2. **始终展示证据和风险** → 即使结构转强，也同时列出「W1 逆风」等风险标签，防止单方面乐观。
3. **用 turning_type 枚举值而非 prob 排序** → 避免用户追问「谁是第一名」。
4. **不进入 State 系统判断** → 「市场转折信号」是独立数据消费，不流入 Agent 辩论或 Decision Ledger。

## 5. 观象指令栏下一步最小命令集

当前三颗胶囊按钮：`我的自选` / `今日转折` / `状态说明`

### 5.1 Phase 2 建议新增 2 颗

| 命令 | 按钮文案 | 发送 prompt |
|------|----------|-------------|
| (已有) | 我的自选 | 我的自选池最近 3 天有哪些状态变化 |
| (已有) | 今日转折 | 列出今天转强早期的标的，只说状态和证据 |
| (已有) | 状态说明 | 解释 D1 zero 和 W1 转强的观察含义 |
| 🆕 | **结构扫描** | 扫描我自选池中 D1/W1 结构出现转强或转弱的标的，列出状态变化和风险 |
| 🆕 | **风险扫描** | 检查我自选池中是否存在 W1/MN1 逆风风险、ADX 衰竭或 BB 带宽异常收缩的标的 |

### 5.2 为什么只加 2 颗

- 5 颗按钮在一行已接近上限，加太多会让指令栏变成按钮海洋。
- 「结构扫描」覆盖转折概率数据 → 用户不需要手动输入复杂条件。
- 「风险扫描」补齐「只看好不看坏」的盲区 → 符合 Research-Only「也告诉你不好的」定位。

### 5.3 占位符提示优化

当前：`例如：000021 / D1 zero / 我的自选池 3 周变化`

建议改为：`输入股票代码、State 条件，或点击下方快捷指令`

理由：更短，引导用户看向下方按钮而非打字。

## 6. 移动端汉堡菜单评估

### 6.1 当前状态

- 窄屏（<860px）下 `.topbar` 切换到 `flex-direction: column`，所有导航链接纵向排列。
- 无折叠/汉堡菜单，6 个顶层链接 + 日期直接暴露。
- 有 viewport meta，有响应式媒体查询，网格和卡片自动适配。

### 6.2 是否 P0？

**不是。** 当前方案可读、可操作，不阻断任何功能。

### 6.3 是否需要立即修？

**建议 Phase 2 做，但不阻塞其他任务。**

理由：
1. 窄屏下导航占约 120-150px 垂直空间，不算灾难但确实挤占首屏。
2. `_top_nav.html` 是独立组件，改它不影响其他页面逻辑。
3. 最小修改量小：一个 CSS media query + 一个 JS toggle。

### 6.4 最小方案（约 30 行改动）

**只改 `_top_nav.html`**，不新增 CSS 文件：

```html
<!-- 窄屏汉堡按钮（仅 <860px 可见） -->
<button class="nav-hamburger" onclick="this.parentElement.classList.toggle('nav-open')"
  style="display:none;">☰</button>

<!-- 现有 nav 链接包在 div 中 -->
<div class="nav-links">
  <a href="/" ...>观察</a>
  ...
</div>
```

CSS 只需 3 条规则：
```css
@media (max-width: 860px) {
  .nav-hamburger { display: block !important; }
  .nav-links { display: none; }
  .nav-open .nav-links { display: flex; flex-direction: column; }
}
```

**不需要改 `web/main.py`、不需要改 `style.css`（可内联在 `_top_nav.html` 的 `<style>` 中）**。

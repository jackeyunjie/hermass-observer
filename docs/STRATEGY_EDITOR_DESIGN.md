# 策略编辑器 + 块式输入 技术方案

版本：v1.0
日期：2026-05-31
借鉴：TradingView（即时反馈）、Notion（块式输入）、Bloomberg（环境归因）

---

## 1. 产品定位

不是让用户写代码。是让用户用**中文自然语言**描述交易规则，系统翻译为条件序列，在 State 底座上做两件事：

- **即时预览**：输入条件即时显示「当前多少只股票满足」
- **环境回测**：跑完整回测并按 MN1/W1 State 环境分层展示收益矩阵

## 3. 条件块设计（借鉴 Notion 块式输入）

### 3.1 可用条件块类型

| 块类型 | 描述 | 参数 |
|--------|------|------|
| **价格突破** | 收盘价 vs 均线/关键位 | 方向（站上/跌破）、均线周期（20/25/50/60）、或自定义值 |
| **成交量** | 成交量 vs 均量 | 比较符（>/<）、均量周期（5/20/50）、倍数 |
| **State 条件** | ef_count / 单周期 State | ef≥(0-3)、MN1 状态、W1 状态、D1 状态、负值过滤 |
| **行业过滤** | 所属 SW 行业 | 包含/排除、行业选择器 |
| **涨跌幅** | 当日涨跌幅 | 区间（-10% ~ +10%） |
| **止损/止盈** | 价格阈值 | 百分比或固定价差 |
| **均线排列** | 多均线顺序 | 短均线 > 长均线 |

### 3.2 块式交互

用户输入方式：
- 直接打字：「价格突破20日线」
- 斜杠命令：`/价格` `/成交量` `/state` `/行业` `/止损`
- 下拉选择：点击「+ 添加条件」→ 弹出条件类型选择器 → 选完类型填参数

**所有块最终翻译为结构化条件 JSON**，不生成 Python 代码、不执行 eval。

## 4. 条件→DuckDB 翻译层

用户看到的条件块 → 后端翻译函数 `_conditions_to_sql(conditions: list[dict]) -> str`

```python
# 用户输入 → 结构化条件
{
    "type": "price_cross",
    "direction": "above",
    "ma_period": 20
}
# → WHERE 子句
"close > AVG(close) OVER (PARTITION BY stock_code ORDER BY date ROWS BETWEEN 19 PRECEDING AND CURRENT ROW)"

{
    "type": "volume_ratio",
    "compare": ">",
    "ma_period": 5,
    "multiplier": 1.5
}
# → WHERE 子句
"volume > 1.5 * AVG(volume) OVER (PARTITION BY stock_code ORDER BY date ROWS BETWEEN 4 PRECEDING AND CURRENT ROW)"

{
    "type": "ef_count",
    "compare": ">=",
    "value": 2
}
# → WHERE 子句
"ef_count >= 2"
```

**翻译层是纯函数，零 LLM 依赖。**

## 5. 即时预览（借鉴 TradingView）

策略编辑器右侧实时显示条件命中数，不需要跑完整回测。

```python
def _condition_preview(conditions: list[dict], date_str: str) -> dict:
    """每个条件块的当前命中数。只查当日数据，秒级返回。"""
    
    for block in conditions:
        sql_where = _conditions_to_sql([block])  # 单条件
        count = duckdb.execute(f"""
            SELECT COUNT(*) FROM d1_perspective_state
            WHERE state_date = '{date_str}' AND ({sql_where})
        """).fetchone()[0]
        block["hits"] = count
    
    # 全部条件合在一起
    total_where = _conditions_to_sql(conditions)
    total_hits = duckdb.execute(f"""
        SELECT COUNT(*) FROM d1_perspective_state
        WHERE state_date = '{date_str}' AND ({total_where})
    """).fetchone()[0]
    
    return {"per_condition": conditions, "total": total_hits}
```

## 7. 数据流

```
用户输入中文条件块
       │
       ▼
前端 events → 条件结构化 JSON
       │
       ├─→ _condition_preview() → 即时命中数显示（<500ms）
       │
       └─→ [运行回测按钮]
              │
              ├─→ _conditions_to_sql() → DuckDB 查询
              ├─→ 回测引擎 run_backtest(custom_conditions)
              ├─→ 结果 × State 底座 → 环境分层矩阵
              └─→ 渲染回测结果卡片
```

## 9. 关键设计决策

| 决策 | 理由 |
|------|------|
| **条件翻译用纯函数，不用 LLM** | 用户的中文输入已经在前端被块化（选类型+填参数），不需要 LLM 做 NER。确定性翻译零幻觉 |
| **即时预览只查当天，不跑回测** | 50ms vs 30s 的体验差异。用户改一个参数立刻看到效果，形成「调试循环」 |
| **环境分层矩阵是核心差异化** | 任何回测工具都能跑总收益——只有 Hermass 能把每个策略按 State 环境拆成 6×2 的矩阵 |
| **不做策略广场 / 不做 Pine Script** | 内测阶段不引入社交功能和不引入编程语言 |

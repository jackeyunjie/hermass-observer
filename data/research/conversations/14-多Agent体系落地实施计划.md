# Hermass 多 Agent 体系落地实施计划

日期：2026-06-01
状态：讨论稿 v1.0
输入：13 份对话文档 + 四模型交叉辩论结果 + 当前代码库现状

---

## 总原则

**不是推翻重建——是在已有生产系统上加层。** 当前 7 个 Agent、p116 Foundation DB、前向观察账本、校准触发器全部保留，作为底座继续运转。新增的是 **Agent 自组织层** 和 **概率推理层**。

**三模型共识的最优混合解**：
- **底座** → Codex 方案（DuckDB 物化视图 + 降级框架）
- **策略层** → Kimi 方案（贝叶斯链 + 场景化仓位公式）
- **架构哲学** → Claude 方案（CAS 涌现 + 五条红线 + 三层权限）
- **接口契约** → Codex + Kimi（JSON Schema + 500ms 截止）

| 2.2 | **节气转折概率回测** | 一次性脚本，不纳入日常 | 历史 3 年 24 节气前后 2 天收益异常检验 | 06-§四.2.2 |
| 2.3 | **事件因子日历** | `config/event_calendar.yaml` | 两会/LPR/政治局/美联储议息/财报季固定日期 | 07-§三 |

### 阶段验收

```
□ 先验概率 = 历史基线 × 时间修正 × 事件修正 × 行业修正 可计算

□ 节气转折检验报告产出（不要求通过，但要产出）
□ 事件日历覆盖 80%+ 的已知重要日期
```

## 第三阶段：收缩观测 Agent 上线（4 周 → 6 周）

### 目标
多周期 BB/枢轴/ATR 三重交叉验证 + 六重突破确认。

### 任务清单

| # | 任务 | 产出 | 验收 | 参考 |
|---|------|------|------|------|
| 3.1 | **BB50/BB100 指标计算** | `timeframe_indicators` 新增列 | 现有数据不回归，新列非空 | 08-§六.2 |
| 3.2 | **枢轴带宽计算** | `pivot_width = (sr_resistance - sr_support) / close` | 三周期均可用 | 08-§二.3 |
| 3.3 | **股票级分位阈值** | per-stock 60/120 bar 历史分位 | 不依赖全市场一刀切 | 08-§三.2 |
| 3.4 | **三重交叉验证逻辑** | 收缩确认 = BB ∧ 枢轴 ∧ ATR 至少二选二 | 可与现有 compression 字段交叉校验 | 08-§二.4 |
| 3.5 | **六重突破确认** | 价格→量能→带宽跳升→ADX→行业→资金流 | 假突破识别率 > 50% | 08-§四.1 |
| 3.6 | **ContractionObserver Agent** | 继承 AgentContext，每日自动运行 | 产出收缩度评分 + 质变事件列表 | 08-§五 |

### 阶段验收

```
□ 每只股票有独立的收缩阈值（不是一刀切）
□ 三重交叉验证产出收缩度 0-100 评分
□ 六重突破确认可识别假突破（回溯3个月历史）
□ ContractionObserver 与 StrategyAdvisor 联动：收缩警报 → 策略优先级 +1
```

## 第四阶段：Agent 协作通信（6 周 → 8 周）

### 目标
Agent 之间能互相发消息、互相校验。

### 任务清单

| # | 任务 | 产出 | 验收 | 参考 |
|---|------|------|------|------|
| 4.1 | **AgentBus 消息总线** | `hermass_platform/bus/agent_bus.py` | pub/sub 模式，延迟 < 500ms | 09-§九.1 |
| 4.2 | **6 个标准消息类型** | contraction_extreme / market_phase_change / false_breakout / weight_adjusted / review_needed / data_stale | 每个类型有 JSON Schema | 09-§七.3 |
| 4.3 | **Agent 互评链路** | 快 Agent ↔ 慢 Agent 每日一致性检查 | 方向分歧度自动计算 | 09-§四.3 |
| 4.4 | **AgentScheduler 自调度** | 按依赖顺序每日自动运行 | 手动触发 → 自动定时 切换 | 09-§七.2 |
| 4.5 | **AgentSupervisor 监控** | 后台进程每分钟检查 Agent 存活 | 宕机自动降级 + 告警 | 09-§十二.3 |

### 阶段验收

```
□ ContractionObserver 发现收缩 → StrategyAdvisor 收到消息（日志可追踪）
□ 所有 Agent 每日按依赖顺序自动运行
□ 一个 Agent 宕机不影响其他 Agent（降级路径生效）
□ AgentSupervisor 能检测到 Agent 宕机并告警
```

## 第五阶段：自进化闭环（8 周 → 12 周）

### 目标
权重自修正 + 因子自发现 + 量变到质变检测。

### 任务清单

| # | 任务 | 产出 | 验收 | 参考 |
|---|------|------|------|------|
| 5.1 | **因子自修正激活** | 指数平滑：新权重 = 旧×0.7 + 实际贡献×0.3 | 仅幅度 ≤20% 自动生效，超限需人类确认 | 07-§8.2 |
| 5.2 | **因子免疫稳态** | 每个因子有 ±10% 耐受带，超带连续 N 日才触发 | 避免过度修正 | 辩论-加分题2 |
| 5.3 | **因子自发现上线** | 异常偏差 → 扫描特征 → 回溯验证 → 候选池 | 第一个自发现因子通过人类审核 | 07-§8.1 |
| 5.4 | **场景库自动构建** | ContractionObserver 从历史中提取 State 组合模式 | 场景模板数 > 20 | 05-§2 |
| 5.5 | **质变检测** | 贝叶斯变点 + Chow 检验 + Simpson 检查 | 20日滚动方向正确率趋势统计显著 | 09-§十三.3 |
| 5.6 | **五条红线硬编码** | 止损/止盈不自动执行、策略结构不可变、数据可信性不自行判断、仓位上限不自动突破、人类可随时拔线 | 红线绕过尝试被代码层拒绝 | Claude 方案 |

### 阶段验收

```
□ 至少一个因子完成了自动权重修正（人类审核通过）
□ 因子免疫稳态机制就位（耐受带生效）
□ 质变检测能区分"真质变"和"统计噪音"（p < 0.05）
□ 五条红线无法被 Agent 自行绕过
```

## 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| AgentMemory schema 设计不当需重构 | 中 | 中 | 阶段 1 先做最小可用设计，允许演化 |
| 节气/事件因子数据不足（历史 < 3 年） | 中 | 低 | 从低权重起步，不强制生效 |
| ContractionObserver 计算量过大拖慢每日流程 | 中 | 高 | Codex 方案：预计算 + 增量窗口 + 只算活跃股票 |
| Agent 自修正脱离人类控制 | 低 | 高 | Claude 五条红线硬编码 + 超 20% 幅度强制人类审核 |
| 阶段并行导致依赖混乱 | 中 | 中 | 严格遵守依赖图，阶段 1 未完成不推进阶段 3 |

## 团队讨论要点

请就以下问题给出反馈：

1. **阶段顺序**：是否同意「底座加固 → 因子体系 → 收缩观测 → Agent 协作 → 自进化」的次序？还是应该调整？

2. **前端优先级**：前端认知卸载（阶段 6）是否应该提前？当前首页 10+ 面板对日常使用的伤害有多大？

3. **收缩观测 vs 因子体系的顺序**：收缩观测需要因子体系做先验概率吗？还是可以先跑 BB/枢轴/ATR 交叉验证再做因子修正？

4. **五条红线的具体定义**：Claude 提的「止损/止盈不可自动执行、策略结构不可变、数据可信性不可自行判断、仓位上限不可自动突破、人类可随时拔线」是否准确？有没有需要增减的？

5. **人力约束**：当前团队有多少人可以投入这个计划？阶段 1 最小需要 1 人 2 周，如果人力更少，优先做哪几个任务？

6. **Qoder 的辩论观点**：Qoder 在四模型辩论中是否提出了和 Codex/Claude/Kimi 实质性不同的方向？如果是，是否需要单独讨论？

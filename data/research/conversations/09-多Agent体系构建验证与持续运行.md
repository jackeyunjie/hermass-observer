# 多 Agent 体系构建、验证与持续运行框架

日期：2026-06-01
主题：多 Agent 体系能否构建、如何验证有效性、如何自组织可持续运行

---

## 一、可行性判断：能构建吗？

### 1.1 已有基础设施盘点

| 能力 | 当前状态 | 文件/模块 |
|------|---------|----------|
| **Agent 基类** | ✅ 已有 `AgentContext` + `AgentResult` | `hermass_platform/agents/base_agent.py` |
| **7 个 Agent 实现** | ✅ 均基于基类，独立运行 | `market_analyst`, `strategy_advisor`, `risk_guardian`, `cognitive_detective`, `coach`, `monetization_butler` |
| **意图路由** | ✅ 关键词匹配将用户请求分发到 Agent | `chat/intent_router.py` |
| **会话管理** | ✅ 多轮上下文保持 + stock_code 记忆 | `chat/conversation_manager.py` |
| **行为账本** | ✅ 用户行为记录 + 统计 | `cognitive/cognitive_ledger.py` |
| **认知画像** | ✅ 基于行为的事件统计分析 | `cognitive/cognitive_profile_builder.py` |
| **前向观察** | ✅ 信号→结果追踪（145+样本积累中） | `forward_observation` |
| **校准触发器** | ✅ 三重门（时间+样本+漂移） | `calibration_trigger.py` |
| **策略验证** | ✅ 回测+前向对比+分层归因 | `simulate_historical_trading`, `strategy_integrity_check` |
| **测试体系** | ✅ 单元/集成/回归/冒烟/smoke 全覆盖 | `tests/` 37 个测试文件 |
| **多周期底座** | ✅ p116_foundation.duckdb | 日更，5510 只股票 |
| **数据新鲜度** | ✅ 自动检测 stale/missing | `web/main.py` freshness_info |

### 1.2 结论：可以构建，但需要分层升级

当前不是从零开始。7 个 Agent 已是生产代码。需要的是从**查询型 Agent（被动响应用户）→ 自组织 Agent（主动监控+自主判断+互相协作）**的升级。

升级分三阶段：

```
阶段1（现在→2周）：Agent 加上记忆和自反馈
    改动：AgentContext 增加 memory_id 字段
          AgentResult 增加 self_assessment 字段
          新增 AgentMemory 存储层

阶段2（2周→1月）：Agent 之间建立通信
    改动：新增 AgentBus（消息总线）
          Agent 可以发布/订阅事件
          新增 InterAgentMessage 数据结构

阶段3（1月→持续）：自组织循环建立
    改动：AgentScheduler 每日自动运行
          评价体系全链路打通
          权重自修正机制激活
```

## 二、如何检查构建的正确性

### 2.1 构建验证金字塔

```
                     ┌──────────────┐
                     │  端到端验证   │ ← 完整链路跑通
                     │  E2E 冒烟    │
                     ├──────────────┤
                     │  集成测试     │ ← Agent间协作+DB读写
                     │  Integration │
                     ├──────────────┤
                     │  单元测试     │ ← 每个Agent独立函数
                     │  Unit        │
                     ├──────────────┤
                     │ 数据契约测试  │ ← data_contract schema
                     │  Contract    │
                     └──────────────┘
```

### 2.2 每层的验证标准

| 层级 | 检查内容 | 工具 | 频率 | 通过标准 |
|------|---------|------|------|---------|
| **契约层** | State hex 编码 bit-exact 一致 | `test_bit_exact.py` | 每次改 State 计算后 | 100% 通过 |
| **单元层** | 每个 Agent 函数输入→输出正确 | `test_*_agent.py` + `pytest` | 每次提交 | 100% 通过 |
| **集成层** | Agent 读取底座 DB 返回结果格式正确 | `test_daily_pipeline.py` | 每日 | AgentResult.status == "ok" |
| **端到端** | 用户问→Router→Agent→回答 完整链路 | `test_e2e_*.py` | 每日/部署前 | HTTP 200 + 回答含预期内容 |

### 2.3 Agent 验收清单

构建新 Agent（如 ContractionObserver）时的强制性验收：

```
□ Agent 继承基类 AgentContext/AgentResult
□ 每次运行写入运行日志（run_log）
□ 每次判断写入判断记录（judgment_record）
□ 有明确的输入数据契约（read from which tables/fields）
□ 有明确的输出数据契约（write to which tables/fields）
□ 有对应的单元测试（覆盖率 ≥ 80%）
□ 有对应的集成测试（读真实DB，不写）
□ Agent 启动后首次运行不需要人工干预
□ 异常数据不导致 Agent 崩溃（catch + log + degrade）
```

## 三、运行时验证：Agent 在做对的事吗？

构建正确 ≠ 运行正确。构建验证回答"Agent 有没有按设计工作"，运行时验证回答"Agent 的判断对市场有没有用"。

### 3.1 运行时验证的四个维度

```
┌────────────────────────────────────────────────────────────┐
│                    运行时验证四象限                         │
│                                                            │
│  Q1: Agent 在产出吗？         Q2: 产出方向对吗？             │
│    ─ 每日运行日志              ─ 次日回溯对比               │
│    ─ 产出数据未空              ─ 方向正确率                 │
│    ─ 延迟在阈值内              ─ 判断→结果偏差              │
│                                                            │
│  Q3: 产出比以前好吗？         Q4: 产出对策略有用吗？         │
│    ─ 滚动窗口对比              ─ 策略胜率是否提升           │
│    ─ 偏差趋势收敛              ─ RR 是否改善               │
│    ─ 因子有效性变化            ─ forward observation 回测   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.2 Q1: Agent 在正常运转吗？（每日检查）

| 检查项 | 方法 | 阈值 | 失败动作 |
|--------|------|------|---------|
| Agent 运行状态 | 检查 run_log 最后时间戳 | < 24h 前 | 告警 |
| 产出数据量 | 检查 Agent 产出表的 row count | > 0 | 告警 + 自动回退到上一日数据 |
| 查询延迟 | 检查 duckdb query 时间 | < 30s | 记录 + 优化 |
| 异常比率 | AgentResult.errors 数量 / 总运行 | < 5% | 超过→降级 |

### 3.3 Q2: 方向对吗？（次日回溯）

这是最重要的验证——Agent 的判断是否正确：

```
每日收盘后，对每个 Agent 的每个判断做回溯：

  判断内容                          回溯方法
  ─────────                        ────────
  "MN1 大概率保持 E/F"             → 次日 MN1 state_score 是否仍 ≥ 14
  "D1 突破概率 70%"                → 次日 D1 close 是否 > SR 阻力
  "行业电子维持强势"                → 次日电子行业 EF2% 是否持平或上升
  "000636 假突破概率 30%"           → 3日后 BB 带宽是否扩张
  "当前市场环境为震荡选择"          → 实际次日涨跌幅度 vs 震荡定义
  "策略 VCP 适配度为最佳适配"        → 20日后实际超额收益 vs 历史均值

回溯结果记录到 AgentMemory 的 accuracy_log 表：
  agent_id | judgment_id | judgment_date | actual_date | direction_correct | strength_deviation
```

### 3.4 Q3: 比以前好吗？（滚动窗口对比）

不是单次判断对不对，而是趋势有没有改善：

```
窗口大小 = 20 个交易日

对比指标：
  方向正确率 = direction_correct_true / total
  平均偏差 = mean(strength_deviation)
  偏差标准差 = std(strength_deviation)

如果 方向正确率在上升 且 平均偏差在下降：
  → Agent 在进化（见识增加）
如果 方向正确率停滞：
  → Agent 已进入平台期，需要引入新因子或新视角
如果 方向正确率下降：
  → 可能是市场范型转移，Agent 需要重新校准
```

## 四、由谁评价

### 4.1 三层评价体系

```
         ┌──────────────┐
         │  人类评价者   │ ← 月度/事件驱动，做架构决策
         │  (架构决策)   │
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │  Agent 互评   │ ← 每周，快慢 Agent 互相校验
         │  (交叉验证)   │
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │  系统自评     │ ← 每日，自动回溯对比
         │  (自动校验)   │
         └──────────────┘
```

### 4.2 系统自评（每日，自动化）

| 谁评价 | 评价对象 | 依据 |
|--------|---------|------|
| **校验 Agent**（新增，专职） | 所有 Agent 的每日判断 | 次日实际数据 vs 昨日判断 |
| **calibration_trigger**（已有） | 策略适配度、State 组合 | 三重门触发校准 |
| **数据新鲜度检查**（已有） | 底座数据时效性 | stale/missing/ok 状态 |

### 4.3 Agent 互评（每周，自动化为主）

| 评价者 | 被评价者 | 评价内容 |
|--------|---------|---------|
| 慢 Agent（MN1） | 快 Agent（D1） | "D1 Agent 本月的方向判断是否与 MN1 大方向一致" |
| 快 Agent（D1） | 慢 Agent（MN1） | "MN1 Agent 的方向判断是否滞后于市场实际变化" |
| MarketAnalyst | StrategyAdvisor | "策略适配建议是否与市场环境描述一致" |
| RiskGuardian | StrategyAdvisor | "建议仓位是否超出当前风控约束" |
| ContractionObserver | 所有策略 Agent | "收缩判断是否被信号验证（突破/假突破）" |

### 4.4 人类评价者（每月，做架构决策）

人评价的不是个体判断的对错，而是**体系层面的问题**：

| 评价层级 | 问题 | 频率 |
|---------|------|------|
| **因子架构** | 当前因子体系是否还需要调整类型框架？ | 月度 |
| **Agent 角色** | 是否需要新增/合并 Agent 角色？ | 月度 |
| **权重校准** | Agent 自修正后的权重人类是否认可？ | 月度（合规性确认） |
| **范型转移** | 市场是否发生了结构性变化，需要重新设定参数框架？ | 事件驱动 |
| **风险边界** | Agent 自修正是否在安全边界内？ | 月度 |

## 六、什么时间评价

### 6.1 评价节奏

```
┌─────────┬──────────────────────────────────────────────────┐
│ 每日     │ 系统自评：方向回溯、数据新鲜度、Agent 状态检查      │
│ 收盘后   │ AgentMemory 写入当日所有判断                      │
│         │ 回溯比对昨日所有判断                               │
│         │ 更新方向正确率滚动窗口                             │
├─────────┼──────────────────────────────────────────────────┤
│ 每周五   │ Agent 互评：快慢 Agent 方向一致性检查               │
│ 收盘后   │ 行业判断 vs 行业实际表现                          │
│         │ 权重偏差检查（是否需要触发修正）                     │
│         │ 新因子候选池更新                                   │
├─────────┼──────────────────────────────────────────────────┤
│ 每月末   │ 全面复盘：人类审阅                                 │
│ 收盘后   │ 月度绩效报告自动生成                               │
│         │ 因子权重修正（Agent提案→人类确认）                   │
│         │ Agent 角色评估（是否需新增/合并）                    │
│         │ 范型转移检测                                       │
│         │ 过拟合检测                                         │
├─────────┼──────────────────────────────────────────────────┤
│ 事件驱动 │ 市场异常日（全市场涨跌 > 3%）                       │
│         │ → Agent 自动标记异常，次日回溯原因                   │
│         │ 重要政策发布日                                     │
│         │ → 事件因子权重临时调整，事后回测                     │
│         │ 数据库重建/底座更新                                 │
│         │ → 全链路回归测试                                   │
└─────────┴──────────────────────────────────────────────────┘
```

### 6.2 为什么是这个节奏

- **每日**：频度最高，但只做自动化、零人工的评价——回溯和状态检查
- **每周**：需要 Agent 之间互相校验，数据积累一周足够看出趋势
- **每月**：人类介入的节点，因为月度是 A 股最基本的周期单位
- **事件驱动**：不走固定日历，重大事件发生时要立即回看

##七、如何自组织

### 7.1 自组织的三层含义

```
层1: 自调度     Agent 自己决定什么时候运行、运行什么
层2: 自协作     Agent 之间自己建立通信、分配子任务
层3: 自进化     Agent 自己调整内部逻辑、权重、甚至角色边界
```

### 7.2 自调度（AgentScheduler）

```
AgentScheduler（每日收盘后自动运行）：

1. 检查底座数据是否就绪（daily_snapshot 存在且 fresh）
2. 按依赖顺序调度 Agent：
   MarketAnalyst → ContractionObserver → StrategyAdvisor → RiskGuardian

3. 每个 Agent 运行完毕后检查返回状态
   - ok → 下一个
   - error → 重试 1 次 → 仍失败则降级处理

4. 全部运行完毕 → 触发自评 Agent → 写入 AgentMemory
5. 如有异常 → 写入异常日志 + 通知人类
```

### 7.3 自协作（AgentBus 消息总线）

```
新增 AgentBus：
  ├── publish(topic, message)     → Agent 发出消息
  ├── subscribe(topic, callback)  → Agent 订阅消息
  └── route(message)              → 系统总线根据消息内容分发

Agent 之间的标准消息类型：

  "contraction_extreme"  → ContractionObserver 发现极致收缩
                        → StrategyAdvisor 订阅，提高该票优先级

  "market_phase_change"  → MarketAnalyst 判断市场阶段转换
                        → 所有 Agent 订阅，重新评估策略适配

  "false_breakout"       → ContractionObserver 标记假突破
                        → RiskGuardian 订阅，暂停开放仓位

  "weight_adjusted"      → 校验 Agent 调整了某个权重
                        → 所有 Agent 订阅，更新内部权重表

  "review_needed"        → 校验 Agent 发现分歧 > 阈值
                        → 人类 + Agent 都需要关注
```

### 7.4 自进化（权重/因子/角色的自我调整）

```
权重自修正（已在 07 文档中定义）：
  → 每月基于实际贡献 vs 设定权重，指数平滑调整

因子自发现（已在 07 文档中定义）：
  → 异常偏差 → 扫描特征 → 回溯验证 → 候选池 → 激活/休眠

角色自调整（新增）：
  如果 Agent A 的某项工作持续被 Agent B 做得更好：
    → 系统检测到"功能重叠且 B 优"
    → 提出合并或重新分工建议
    → 人类确认后执行
```

## 八、如何记忆

### 8.1 当前已有的记忆

| 记忆类型 | 当前实现 | 记忆什么 |
|---------|---------|---------|
| 用户认知记忆 | `cognitive_ledger.py` | 用户行为事件 + 画像 |
| 会话记忆 | `conversation_manager.py` | 多轮对话上下文 |
| 前向观察记忆 | `forward_observation` | 信号→结果追踪 |

### 8.2 需要新增的 Agent 记忆

```
outputs/agent_memory/
├── AgentMemory.duckdb
│   ├── agent_judgments        # Agent 每次判断的完整记录
│   │   ├── agent_id
│   │   ├── judgment_id
│   │   ├── judgment_date
│   │   ├── judgment_type      # direction/strength/timing/probability
│   │   ├── judgment_content   # JSON: 判断的完整内容
│   │   ├── confidence
│   │   ├── factors_used        # 用了哪些因子, 各自的权重
│   │   └── context_snapshot    # 判断时的市场环境快照
│   │
│   ├── judgment_outcomes       # 回溯结果（次日/后验证）
│   │   ├── judgment_id (FK)
│   │   ├── actual_date
│   │   ├── actual_value
│   │   ├── direction_correct
│   │   ├── strength_deviation
│   │   └── scenario_label     # 属于哪个场景（contraction_extreme 等）
│   │
│   ├── agent_scenario_library  # 场景库（见识积累）
│   │   ├── scenario_id
│   │   ├── scenario_name       # "三周期窒息收缩后突破"
│   │   ├── features            # State组合+BB带宽+枢轴带宽+ATR
│   │   ├── sample_count
│   │   ├── true_breakout_rate
│   │   ├── avg_excess_return
│   │   └── last_updated
│   │
│   ├── factor_weights_history  # 权重修正历史
│   │   ├── factor_name
│   │   ├── date
│   │   ├── old_weight
│   │   ├── new_weight
│   │   ├── actual_contribution
│   │   └── reason
│   │
│   └── agent_evolution_log     # Agent 自身成长记录
│       ├── agent_id
│       ├── date
│       ├── direction_accuracy_20d
│       ├── avg_deviation_20d
│       ├── scenarios_discovered
│       └── evolution_stage     # "growing"/"stable"/"plateau"/"degenerating"
```

### 8.3 记忆的运行方式

```
每次 Agent 做出判断 → 写入 agent_judgments
次日收盘后 → 校验 Agent 对比实际 → 写入 judgment_outcomes
每周五 → 更新 agent_scenario_library（场景样本库更新）
每月 → factor_weights_history 新行 + agent_evolution_log 新行
```

## 九、如何互相沟通

### 9.1 沟通的两条通道

| 通道 | 用途 | 结构 |
|------|------|------|
| **AgentBus（实时）** | Agent 之间的即时消息 | pub/sub 模式 |
| **共享数据库（异步）** | Agent 之间共享判断和状态 | 读写 AgentMemory |

### 9.2 AgentBus 消息格式

```python
@dataclass
class InterAgentMessage:
    message_id: str
    from_agent: str
    to_agent: str | None      # None = 广播
    topic: str
    priority: int              # 0=常规, 1=重要, 2=紧急
    payload: dict
    timestamp: str
    requires_response: bool
    response_deadline: str | None
```

### 9.3 标准通信场景

| 场景 | 发出方 | 接收方 | 消息内容 |
|------|--------|--------|---------|
| 收缩警报 | ContractionObserver | StrategyAdvisor, RiskGuardian | 某股票达到极致收缩，建议提高关注度 |
| 市场阶段转换 | MarketAnalyst | 全部 Agent | 市场从"震荡"切换为"趋势"，请各自调整内部参数 |
| 策略适配异常 | StrategyAdvisor | MarketAnalyst | 策略信号大幅增加但市场环境未变化，请检查 |
| 假突破预警 | ContractionObserver | RiskGuardian | 某股票标记为假突破，暂停开放仓位建议 |
| 权重修正提案 | 校验 Agent | 全部 Agent + 人类 | 某因子权重建议从 X 调整为 Y |
| 范型转移嫌疑 | 校验 Agent | 全部 Agent + 人类 | 连续多日全部 Agent 判断偏差 > 阈值 |

## 十一、如何改进

### 11.1 改进的触发方式

| 触发条件 | 动作 |
|---------|------|
| 方向正确率连续 5 日下降 | → 自动触发 Agent re-calibration |
| 单个因子贡献度 < 10% 预期 | → 自动降权，列为候选休眠 |
| 新因子候选池满足激活条件（20日持续有效） | → 自动提案加入因子体系 |
| Agent 互评发现分歧 > 阈值 | → 标记为需要人类审阅 |
| 假突破识别率 < 30% | → ContractionObserver 调整质变确认参数 |
| 人类发现 Agent 系统性盲点 | → 手动添加因子类型框架 → Agent 自行填充细节 |

### 11.2 改进的范围边界

| Agent 可自行改进的 | 需要人类确认的 |
|------------------|-------------|
| 因子权重（±20%以内） | 因子权重（变化 > 20%） |
| 阈值微调（如 Q20 → Q15） | 新增因子类型框架 |
| 场景库自动更新 | Agent 角色合并/分拆 |
| 假突破判断参数 | 风险边界修改 |
| 运行频率微调 | 新 Agent 上线 |

### 11.3 改进的闭环验证

```
改进不是"改了就行"——改了必须验证：

  改进执行 → 观察窗口（5-20日）
              │
              ├── 方向正确率 ↑ 或偏差 ↓ → 改进有效 → 固化
              ├── 方向正确率 → → 改进无效 → 回滚 → 重新分析
              └── 方向正确率 ↓   → 改进有害 → 紧急回滚

每次改进（包括 Agent 自改进）都要写入 evolution_log
→ 这样才能追踪"Agent 是如何一步步变聪明的"
```

## 十二、如何持续运行

### 12.1 核心原则：Assume Failure

任何一个 Agent 都可能在某一天因为数据问题、异常市场、代码 bug 而失败或输出错误判断。体系的持续运行必须建立在这个前提上。

```
单 Agent 失败       体系的行为
─────────────      ──────────────────────────────────
D1 Agent 宕机      其他 Agent 继续运行，标注"D1 视角暂缺"
ContractionObserver 失败   默认当天不产生收缩信号，等待次日恢复
校验 Agent 宕机     其他 Agent 仍然产出判断，但标注"未校验"
底座数据 stale      所有 Agent 降级运行，输出置信度下调
```

### 12.2 降级路径

```
正常运行 → Agent 全量判断，置信度正常
  │
数据延迟（stale < 阈值）
  │ → Agent 仍产出判断，置信度 -10%
  │ → 标注"数据时效性降低"，等数据恢复后回溯修正
  │
数据严重延迟（stale > 阈值）
  │ → Agent 暂停基于该数据的判断
  │ → 使用上一日判断的延续作为"占位"
  │ → 告警人类
  │
Agent 自身异常
  │ → 该 Agent 输出标记为"unavailable"
  │ → 其他依赖它的 Agent 使用该 Agent 上一日输出作为代替
  │ → 或跳过依赖该 Agent 的判断步骤
  │
全链路故障
  │ → 回退到纯底座输出模式
  │ → 前端降级为"底层数据展示"（不显示 Agent 判断）
  │ → 告警人类，等修复后恢复
```

### 12.3 运行监控

新增 `AgentSupervisor` 进程，持续监控：

```
AgentSupervisor（后台进程，每分钟检查一次）：
  ├── 检查各 Agent 进程是否存活
  ├── 检查 AgentSleep（数据库连接）是否正常
  ├── 检查各 Agent 最后输出时间是否在正常窗口内
  ├── 检查是否有 Agent 输出异常增大（重复判断/无限循环）
  └── 异常时自动执行降级策略
```

## 十三、如何量变到质变

### 13.1 量变是什么在积累

| 量变维度 | 积累内容 | 积累速度 |
|---------|---------|---------|
| 场景样本 | 每个交易日 × 5510 只股票 → 新的场景实例 | 高速 |
| 判断记录 | 每个 Agent 每天产生 N 条判断 | 中速 |
| 因子验证 | 因子权重每周校准一次 | 低速 |
| 见识增长 | 场景库中有效模板数量 | 中速 |

### 13.2 质变的触发条件

```
质变不是人定义的——质变是在量变积累到一定程度后，系统呈现出的新性质：

质变1: Agent 开始发现人未设定的新因子
  触发条件: 场景库积累 > 500 模板 + 判断记录 > 10000 条
  表现: Agent 发现"期权交割日前一天 D1 score 平均偏离 2.1σ"
         → 这是人没想过要检查的因子

质变2: 方向正确率突破平台期
  触发条件: 因子权重持续优化 ≥ 3 个完整周期
  表现: 方向正确率从 55-60% 跳升到 65%+

质变3: Agent 之间形成"互补判断"而非"重复确认"
  触发条件: 多 Agent 互评 ≥ 5 周
  表现: 快 Agent 和慢 Agent 的判断分歧比趋于稳定，
        且分歧信息本身成为有效信号
        （分歧越大 → 市场转折概率越高）

质变4: 系统不再需要人的日常干预
  触发条件: 连续 3 个月人类零干预，Agent 正常运行
  表现: 人类角色从"操作者"转变为"监督者"
        → 这正是"将能而君不御"的实现
```

### 13.3 质变如何确认

人不能靠"感觉变好了"来判断质变——要用统计检验：

```
假设检验：质变前后，Agent 的方向正确率是否有显著提升？

  原假设 H0: μ_after = μ_before（质变前后无区别）
  备择 H1:  μ_after > μ_before（质变后有提升）

  用独立样本 t-test 或 Bootstrap 重抽样
  显著性水平 α = 0.05

  如果 p < 0.05 → 质变确认
  如果 p ≥ 0.05 → 可能是噪音，继续观察

同样检验：
  质变前后 RR 改善 → Mann-Whitney U（非正态分布）
  质变前后假突破识别率 → Fisher's exact test
```

## 结语

这个体系不是要一次性建完。从当前代码库出发，分三步走：

**第一步**（立即可做）：AgentMemory 数据库 + 每日判断记录 + 次日回溯
**第二步**（需要 1-2 周开发）：AgentBus 消息总线 + 互评 + 自调度
**第三步**（运转中自然到达）：因子自修正 + 场景库积累 + 量变到质变

**最重要的原则**：不是等一切完美再上线，而是让系统先跑起来，在运转中改进。道德经第 64 章说"合抱之木，生于毫末"——Agent 体系的质变，从每一次判断记录开始积累。

# 自组织多 Agent 运营架构

日期：2026-06-01
主题：多 Agent 集体智慧 + 多周期运转底座 → 强化学习与自组织运营

## 核心陈述

> "我们不是全部推翻之前的，资金流也还是在 D1 Agent 内部保留。我们是先验概率加是否天花板受压制，压制的概率多少，突破后如何跟踪等等一系列的场景化 AI 应用流。仅仅靠我提前预设无法应对市场的长期运转，我们靠多 Agent 的集体智慧和我们多周期运转底座结合，同时各 Agent 们自我成长——记忆的成长、见识的成长、稳定性和创新性的成长，来完成强化学习和自组织运营来实现。"

## 关键理解

### 1. 不推翻，而是升级

当前系统不是错的，只是处于"预设规则"阶段：
- D1 Agent 内部保留资金流 → ✓ 继续保留
- State 编码 + EF 筛选 + 策略信号 → ✓ 继续运行
- 行业轮动 + 前向观察 → ✓ 继续积累

但需要往上加一层：**Agent 层**，让 Agent 来解读这些数据，而不是数据直接输出结论。

### 2. "场景化 AI 应用流" 的含义

不是简单的 `if ef_count >= 2 → 推荐`，而是场景驱动的概率推理链：

```
场景：天花板压制检测
  ├── 先验概率：历史类似 State 组合下被压制的概率
  ├── 当前压制因子：SR 阻力距离 + 量能萎缩程度 + 行业 ETF State
  ├── Agent 判断：被压制概率是多少
  └── 输出：不是"不能做"，而是"突破概率 XX%，压制概率 XX%"

场景：突破后跟踪
  ├── 突破确认：假突破概率（基于历史 State 转移数据）
  ├── 跟踪指标：D1 ADX 变化 + 资金流确认 + 板块承接
  ├── Agent 持续观察：不是一次性判断，是持续跟踪
  └── 输出：当前处于突破第 X 天，延续概率 XX%，衰竭信号是否出现
```

### 3. Agent 自我成长的四个维度

| 维度 | 含义 | 当前系统对应 | 需要补的 |
|------|------|------------|---------|
| **记忆** | Agent 记住历史判断和结果 | cognitive_ledger 记录用户行为 | 需要 Agent 自己的判断→结果记忆链 |
| **见识** | Agent 见过的市场场景越来越多 | forward_observation 积累中 | 需要按场景归类，形成场景库 |
| **稳定性** | 相同场景下判断的一致性 | 无 | 需要 Agent 自我校验机制 |
| **创新性** | 发现新模式、新关联 | 无 | 需要 Agent 能提出假设并验证 |

## 现有 Agent 体系盘点

当前 `hermass_platform/agents/` 已有 7 个 Agent：

| Agent | 当前职能 | 自组织升级方向 |
|-------|---------|-------------|
| **market_analyst** | 查询市场环境数据 | 升级为：基于多周期运转底座，自主判断市场阶段并跟踪阶段转移 |
| **strategy_advisor** | 查询策略适配度 | 升级为：自主评估各策略在当前市场态下的预期质量，并持续校准 |
| **risk_guardian** | 查询风控参数 | 升级为：基于场景（压制/突破/延展）动态调整风控参数 |
| **coach** | 静态知识库回答 | 升级为：根据用户行为和 Agent 判断历史，个性化指导路径 |
| **cognitive_detective** | 构建用户认知画像 | 升级为：同时构建"Agent 自己的认知画像"——Agent 也需要成长 |
| **monetization_butler** | 订阅查询 | 保持，非核心运转层 |
| **base_agent** | Agent 基类 | 升级为：增加 self_learning 接口 |

### 自组织循环

```text
┌─────────────────────────────────────────────────────┐
│                   每日运转循环                         │
│                                                     │
│  1. 底座产出 State 矩阵 + 策略信号 + 资金流            │

```

## 当前差距：从"查询型 Agent"到"自组织 Agent"

| 能力 | 当前状态 | 需要 |
|------|---------|------|
| Agent 查询数据库 | ✓ 已有 (duckdb) | - |
| Agent 输出结构化结果 | ✓ 已有 (AgentResult) | - |
| Agent 记住自己的判断 | ✗ | 需要：判断→结果 反馈链 |
| Agent 比较判断 vs 实际 | ✗ | 需要：forward_observation 接入 Agent |
| Agent 自主调整权重 | ✗ | 需要：场景库 + 后验统计 |
| Agent 之间协作 | ✗ 部分(通过 Router) | 需要：多 Agent 联合推理 |
| Agent 发现新模式 | ✗ | 需要：异常检测 + 假设生成 |

## 渐进路径

### 第一步：给 Agent 加上记忆（最小成本）
- 每次 Agent 产生判断（如"SR 阻力压制概率 70%"），记录到 AgentMemory
- 下一日底座更新后，自动回溯昨日判断是否正确
- Agent 看到自己的"判断→结果"记录

### 第二步：场景库构建
- 从 forward_observation 中提取典型场景模板：
  - "MN1=E, W1=E, D1=D, SR 刚突破" → 这是"突破待确认"场景
  - "MN1=E, W1=F, D1=F, 资金流背离" → 这是"高位延展+背离"场景
- 每个场景积累历史统计

### 第三步：概率化输出
- 不再输出"推荐/不推荐"
- 输出"在当前场景下，历史类似情况 N 次，突破延续概率 XX%，压制回落概率 XX%"
- 首页展示的不是结论，是概率分布

### 第四步：Agent 协作
- 市场 Analyst 说"当前市场环境为震荡选择"
- 策略 Advisor 根据这个环境，给出各策略的预期胜率
- 风控 Guardian 根据策略预期，调整仓位建议
- 三个 Agent 不是独立回答，是接力推理

### 第五步：强化学习闭环
- Agent 的判断和实际结果自动对比
- 当判断持续偏离实际 → Agent 自动调低该场景的置信度
- 当新模式出现 → Agent 标记为"需要学习"

## 与原对话的衔接

此前的权重框架（D1 50-60% / W1 20-25% / 行业 15-25% / MN1 10-15%）仍然是正确的——它描述了**底座层面的信息权重**。但 Agent 层应该在这个底座之上做概率推理，而不是重复底座的结论。

## 一、评价节奏调整

### 原方案（09文档）→ 新方案

| 维度 | 原节奏 | 新节奏 | 理由 |
|------|--------|--------|------|
| AI 自评 | 每日收盘后 | **每 4 小时** | A股开盘4h内足以让 Agent 形成判断并看到部分结果 |
| Agent 互评 | 每周五 | **每日**（收盘后） | 快慢 Agent 每天就应该互相对齐 |
| 人类评价 | 每月 | **至少每日一次** | 人需要每天确认 Agent 是否在轨道上 |

### 4 小时自评的具体执行

```
交易日 09:30 开盘
    │
10:30 → 第一轮自评（盘中）：Agent 盘中判断回顾
    │   - 检查开盘1h的方向判断是否正确
    │   - 更新短期方向正确率
    │
14:30 → 第二轮自评（收盘前）：盘中判断汇总
    │   - 对比上午判断 vs 下午实际走势
    │
15:30 → 第三轮自评（收盘后）：全天回溯
    │   - 完整回溯所有 Agent 的今日判断
    │   - 对比昨日判断 vs 今日实际
    │   - 写入 AgentMemory
    │
次日 09:00 → 盘前自评：数据新鲜度检查 + 底座状态
    
非交易日：每4小时一次数据新鲜度检查
```

### 每日人类评价模板

```
人类每日审阅清单（自动生成，人只需5分钟过一遍）：

  □ 今日 Agent 方向正确率趋势（正常 / 异常 / 严重异常）
  □ 是否有标记为"紧急"的 Agent 异常（假突破预警/权重崩塌/范型转移嫌疑）
  □ 是否有 Agent 擅自调整的权重需要人类确认
  □ 今日判断失误最大的 3 只股票（自动列出）
  □ 今日最值得关注的 3 个信号（自动列出）
  
  人只需要在异常项上做决策，正常项默认通过。
```

| 属性 | 内容 |

### 2.2 核心能力（与 Hermass 相关）

| 能力 | 详情 | Hermass 可借鉴度 |
|------|------|----------------|
| **Closed Learning Loop** | Agent 从经验中创建 Skill，使用中自我改进，定期 nudge 自己持久化知识 | ⭐⭐⭐⭐⭐ |
| **Skill 系统** | Agent 完成复杂任务后自动创建 Skill，Skill 使用中自我改进 | ⭐⭐⭐⭐⭐ |

| **MCP 集成** | MCP 工具客户端 | ⭐⭐⭐⭐ |

### 2.3 架构亮点

**Skill 自创建机制**（最值得借鉴）：

```
Agent 完成复杂任务
    → 检测到任务中包含可复用的模式
    → 自动创建 Skill（skill.yaml + SKILL.md）
    → 下次遇到类似任务时自动加载对应 Skill
    → Skill 在每次使用中自我改进
```

**这与 Hermass 的 ContractionObserver 场景高度对应**：

- CcontractionObserver 发现"三周期极致收缩后突破" → 自动创建该场景的 Skill 模板
- 下次遇到相同收缩模式 → 自动加载该 Skill，包含历史胜率、假突破率、最优策略
- Skill 在每次使用中更新统计数据

**子 Agent 委托模型**：

```python
# hermes-agent 的 delegate 工具
# 主 Agent 可以 spawn 一个隔离子 Agent 执行特定任务
# 子 Agent 有独立的工具集和上下文窗口

# Hermass 对应场景：
# 主 Agent 发现某行业 10 只股票同时收缩
# → spawn 10 个子 Agent，各自分析一只
# → 汇总结果，形成行业级别判断
```

### 2.4 Hermass 可直接采用的部分

| 功能 | hermes-agent 实现 | Hermass 实现方式 |
|------|------------------|-----------------|
| **Skill 自动创建** | `skills/` 目录自动生成 | `outputs/agent_skills/` 目录，场景模板自动生成 |
| **FTS5 搜索** | `hermes_state.py` SessionDB | AgentMemory.duckdb 加 FTS5 扩展 |
| **Cron 调度** | `cron/` 子模块 | 已有 hermes_cron.json → 扩展到 Agent 自调度 |
| **子 Agent 委托** | `delegate_tool.py` | 行业扫描 / 批量个股分析 并发 |
| **多平台通知** | gateway/ 跨平台 | 企业微信/飞书/邮件 通知 Agent 异常 |

## 三、rohitg00/agentmemory 深度分析

### 3.1 项目概况

| 属性 | 内容 |
|------|------|
| 项目 | rohitg00/agentmemory |
| 性质 | 编码 Agent 持久化记忆引擎 + MCP 服务器 |
| 语言 | TypeScript（Node.js） |
| 版本 | v0.9.4（2026-04-29） |
| 核心思想 | 基于 Karpathy 的 LLM Wiki 模式，加入置信度评分、生命周期、知识图谱 |

### 3.2 核心技术特征

**三级记忆模型**：

```
Working Memory (session)        → 当前对话上下文
    ├── 自动消退（会话结束）
    ▼
Episodic Memory (session log)   → 每次对话的摘要记录
    ├── 压缩存为 JSON + embedding
    ▼
Semantic Memory (knowledge)     → 持久化知识条目
    ├── 置信度评分（用户确认 > Agent 推断）
    ├── 生命周期（Temporary → Stable → Canonical → Archived）
    └── 知识图谱（实体关系）
```

**混合检索**：BM25 + Vector (all-MiniLM-L6-v2) + Graph + RRF 融合

**LongMemEval-S 基准**：R@5 = 95.2%，MRR = 88.2%（行业最佳）

**自动捕获**：12 个钩子（PreToolUse/PostToolUse/Stop/SubagentStop 等），自动捕获 Agent 行为

### 3.3 Hermass 可直接采用的部分

| 功能 | agentmemory 实现 | Hermass 应用 |
|------|-----------------|-------------|
| **置信度评分** | 用户确认 > Agent 推断 > 默认 | Agent 判断置信度 × 人类确认 × 历史验证 |
| **生命周期管理** | Temporary→Stable→Canonical→Archived | 场景因子从候选→验证→激活→休眠（已在07文档定义，结构与agentmemory吻合） |
| **混合检索** | BM25+Vector+Graph+RRF | AgentMemory 的 judgment 检索 |
| **知识图谱** | 实体+关系图 | 股票→行业→State组合→策略信号的关系网络 |
| **MCP 集成** | 标准 MCP 协议 | Agent 间通信标准化（可替代自建 AgentBus） |

### 3.4 与 AgentMemory 设计的对照

agentmemory 的四级生命周期与我们已在07文档中定义的因子池生命周期完全一致：

| agentmemory | Hermass 07文档 | 一致度 |
|-------------|---------------|--------|
| Temporary | proposed（发现期） | ✅ 一致 |
| Stable | validating（验证期） | ✅ 一致 |
| Canonical | active（激活期） | ✅ 一致 |
| Archived | dormant + archived | ✅ 一致 |

说明 Hermass 的设计方向与业界最佳实践对齐。

## 四、Harness Engineering 方法论深度分析

### 4.1 核心定义

**Harness Engineering** 是 OpenAI 团队在构建内部产品中的实践经验提炼，核心理念：

> **0 行手写代码。** 人设计约束和反馈循环，Agent 执行和维护代码。

由 Intense-Visions 社区标准化为完整的方法论框架。

### 4.2 七大核心原则

| # | 原则 | 含义 | Hermass 映射 |
|---|------|------|-------------|
| 1 | **Context Engineering** | 所有知识存 git，Agent 可读取 | 09文档的 AgentMemory + 本次的 Obsidian wiki 方案 |
| 2 | **Architectural Rigidity** | 机械约束（linter/type-check）强制执行，不让 Agent 靠自觉 | 数据契约层 + JSON schema + bit-exact 测试 |
| 3 | **Agent Feedback Loop** | Agent 自审→同行审→人审 | 09文档的三层评价体系（系统自评→Agent互评→人类评价） |
| 4 | **Entropy Management** | 系统性清理防止技术债累积 | 因子生命周期（激活→休眠→归档）+ 定期 lint |
| 5 | **Depth-First Implementation** | 完整做完一个功能再做下一个 | 先做 AgentMemory，再做 AgentBus，最后自组织 |
| 6 | **Measurable Success** | 量化 Agent 自主性、覆盖率 | 09文档的核心 KPI 体系 |
| 7 | **Deterministic-vs-LLM** | 能用 if-else 就不要用 LLM 判断 | State 计算（确定性）≠ Agent 判断（概率性）|

### 4.3 最核心的启示

**"Humans steer. Agents execute."**

Harness Engineering 的核心理念完美对应了孙子的"将能而君不御"：

> 人（君主）的角色是设计环境、指定意图、建立反馈循环
> Agent（将领）的角色是在约束内自主执行、自我审查、自我改进

**对 Hermass 的直接应用**：

```
人类工程师                       Hermass Agent
──────────                      ─────────────
定义因子类型框架                Agent 自发现具体因子实例
定义校准最小周期（月）          Agent 自主调整具体权重数值
定义降级路径和风险边界          Agent 自主降级和恢复
审阅月度偏离报告                Agent 每天的判断自我回溯
不做日常干预                    Agent 自调度 + 自协作
```

### 4.4 "Repository Knowledge is System of Record" 的直接应用

Harness Engineering 的 AGENTS.md 达到了 1046 行——整个项目的知识地图都在 git 中。

**Hermass 对应方案**：将数据/分析/设计决策以 Obsidian wiki 形式存入 `data/research/conversations/` 和未来的 `docs/wiki/`，Agent 可以直接读取。

这正是 Karpathy 提出的三层架构可以直接映射到 Hermass：

```
Karpathy 三层               Hermass 对应
────────────                ────────────
Layer 1: Raw Sources        outputs/ 原始数据（duckdb, json, csv）
Layer 2: The Wiki           data/research/conversations/ + docs/wiki/
Layer 3: The Schema         AGENTS.md + config/
```

## 五、渐进式披露 MD + Obsidian 结构化记忆

### 5.1 Karpathy 的 Wiki 模式

Karpathy 的核心思路：**不要让 LLM 每次都从原始文档检索——让它持续维护一个结构化的 Wiki。**

四个核心操作：

| 操作 | 功能 | Hermass 对应 |
|------|------|-------------|
| **Ingest** | 新数据进入时，LLM 更新 Wiki 中的相关页面 | 每日底座产出 → Agent 更新场景库和 State 分布摘要 |
| **Query** | 对着 Wiki 提问，好回答回存 Wiki | 用户问市场环境 → Agent 生成的回答存入场景库 |
| **Lint** | 定期检查 Wiki 健康度（矛盾、过时、孤儿页面） | Agent 月度复盘：检查因子是否过时、场景是否失活 |
| **Log** | 操作日志，追踪 Wiki 变化 | agent_evolution_log（已在09文档定义） |

### 5.2 Obsidian 结构化记忆在 Hermass 的应用

Obsidian 的核心优势：**wikilink + 图形化知识图谱 + 本地 Markdown**。

**Hermass 的具体应用方案**：

```
docs/wiki/
├── 00-INDEX.md                # 全局索引（自动生成）
├── AGENTS.md                  # Agent 操作手册（链接到 wiki）
├── state/
│   ├── State编码体系.md        # State score → hex 完整说明
│   ├── E-F共振策略.md          # EF 筛选的完整逻辑
│   └── 三周期权重分配.md        # D1/W1/MN1 权重分析
├── strategies/
│   ├── VCP收缩突破.md
│   ├── 2560趋势跟踪.md
│   ├── 布林强盗延展.md
│   └── 策略适配度计算.md
├── agents/
│   ├── Agent角色定义.md        # 所有Agent的能力边界
│   ├── 自组织机制.md
│   ├── 记忆模型设计.md
│   └── 评价体系.md
├── scenarios/
│   ├── 收缩极致→突破.md        # 场景模板（Agent自动创建）
│   ├── 三周期共振突破.md
│   ├── 假突破识别.md
│   └── 高位延展背离.md
├── factors/
│   ├── 时间因子库.md
│   ├── 事件因子库.md
│   └── 因子权重演进历史.md
├── operations/
│   ├── 每日运行SOP.md
│   ├── 降级路径.md
│   └── 每日复盘模板.md
└── glossary.md                 # 术语表（Agent和人类共用）
```

### 5.3 渐进式披露原则

Agent 在读取 Wiki 时遵循"渐进式披露"原则——先读索引，需要时才深入：

```
1. 先读 00-INDEX.md → 了解有什么
2. 再读 glossary.md → 理解术语
3. 需要具体知识时 → 打开对应页面
4. 不会一次性加载整个 Wiki → 节省 token
```

这与 agentmemory 的 `ContextSummary.md` 方案异曲同工。

## 六、综合适用性评估

### 6.1 优先级矩阵

| 技术/方案 | 对 Hermass 的适用度 | 实现难度 | 优先级 | 建议动作 |
|----------|-------------------|---------|--------|---------|
| **4小时自评节奏** | ⭐⭐⭐⭐⭐ | 低 | **P0 立即** | 直接调整评价节奏配置 |
| **Harness Engineering 原则** | ⭐⭐⭐⭐⭐ | 低（方法论层面） | **P0 立即** | 将七大原则写入 AGENTS.md |
| **Karpathy Wiki 三层架构** | ⭐⭐⭐⭐⭐ | 中 | **P1 本周** | 建立 docs/wiki/ 目录结构 |
| **agentmemory 混合检索** | ⭐⭐⭐⭐ | 中 | **P1 本月** | AgentMemory 加 FTS5 + embedding |
| **hermes-agent Skill 系统** | ⭐⭐⭐⭐⭐ | 高 | **P2 下月** | 场景模板自动创建与改进 |
| **hermes-agent 子Agent委托** | ⭐⭐⭐⭐ | 中 | **P2 下月** | 行业批量分析并发 |
| **hermes-agent Cron调度** | ⭐⭐⭐ | 低 | **P2** | 已有 hermes_cron.json 扩展 |
| **agentmemory MCP 集成** | ⭐⭐⭐ | 中 | **P3** | 探索替代自建 AgentBus |

### 6.2 需要警惕的不适用点

| 技术 | 不适合 Hermass 的原因 |
|------|---------------------|
| hermes-agent 的多后端运行 | Hermass 已有自己的部署体系（服务器+systemd），不需新增后端 |
| agentmemory 的 MCP 作为 AgentBus | Hermass Agent 间的通信更复杂（需要传递状态矩阵，不只是文本消息），自建 AgentBus 更适合 |
| Harness Engineering 的 ESLint 约束 | Hermass 是 Python 项目，用的是 ruff + mypy |

### 6.3 三阶段立即行动

```
第一阶段（今天-本周）：框架层面
  ├── ✅ 评价节奏调为 4hAI自评 + 每日人类评价
  ├── ✅ Harness Engineering 七大原则写入 AGENTS.md
  └── 📋 建立 docs/wiki/ 目录骨架（从现有 conversations/ 提取）

  ├── 📋 知识图谱（股票→行业→State→策略信号）
  └── 📋 agentmemory 的置信度评分和生命周期模型

第三阶段（下月）：Agent 能力层面
```

##七、Human-in-the-Loop 的新形态

将 Harness Engineering + 渐进式披露 + 4小时节奏结合后，Hermass 的人类角色发生了根本变化：

```
         旧角色                           新角色
    ─────────────                   ─────────────
    每天看数据、做判断              每天审阅 5 分钟自动生成的清单
    手动调整策略参数                Agent 提案调整，人确认/否决
    担心遗漏重要信息                Wiki 自动 lint 标记矛盾/过时
    凭感觉评价 Agent               KPI 仪表板量化评价
    不了解 Agent 在想什么           Wiki 透明展示 Agent 的判断逻辑

    操作者  ──────────────────→  监督者 + 架构决策者
    "将能而君不御" —— 孙子兵法第五条
```

**本质上，人的角色从"执行交易的操盘手"变成了"多 Agent 运转系统的架构师"。**
